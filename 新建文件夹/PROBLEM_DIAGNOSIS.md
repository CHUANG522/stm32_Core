## 🎯 真正的问题：【上位机接收端的问题】

上位机软件可能有以下问题：

### 🚨 **问题A：串口缓冲区问题**
- 上位机打开串口后没有清空缓冲区
- 之前遗留的数据污染了新接收

### 🚨 **问题B：接收超时**
- 上位机设定了接收超时，但单片机响应太慢
- 或者上位机没有等待足够的时间

### 🚨 **问题C：波特率不匹配**
- 虽然都设为19200，但可能有时序问题
- RS485收发切换延迟不足

### 🚨 **问题D：帧头帧尾识别**
- 上位机在查找帧头/帧尾时的逻辑有问题
- 可能在错误的位置识别到"A5B6"或"B6A5"

## 💡 我发现的真实问题！

看一下你上面上传的图片，我看到：

```
...compiling Protocol.c...
```

单片机在 **发送** 这些帧时，关键问题可能是：

### **【真正的根本原因】**

1. **第一次通信成功** → 缓冲区还空着，帧格式正确
2. **第二次通信失败** → 第一帧的残留数据仍在缓冲区

**症状：**
- 上位机：发送第一条命令 → 收到响应 ✅
- 上位机：发送第二条命令 → 收到的是第一条的重复数据 ❌

**根本原因：**
```c
/* Protocol_SendFrameASCII() 中没有清空发送缓冲区！*/
void Protocol_SendFrameASCII(uint8_t *frame_bin, uint16_t frame_len)
{
    char ascii_buf[512];
    for (i = 0; i < frame_len; i++) {
        sprintf(&ascii_buf[i*2], "%02X", frame_bin[i]);
    }
    RS485_SendData((uint8_t*)ascii_buf, frame_len * 2);
    // ❌ 缺少：等待发送完成 + RS485切换回接收模式
}
```

---

## ✅ 解决方案

需要在 Protocol.c 的发送后添加：

```c
void Protocol_SendFrameASCII(uint8_t *frame_bin, uint16_t frame_len)
{
    char ascii_buf[512];
    uint16_t i;
    
    for (i = 0; i < frame_len; i++) {
        sprintf(&ascii_buf[i*2], "%02X", frame_bin[i]);
    }
    
    RS485_SendData((uint8_t*)ascii_buf, frame_len * 2);
    
    /* ✅ 添加这部分 */
    delay_1ms(100);  /* 等待发送完全完成 */
    
    /* 确保RS485已切回接收模式 */
    RS485_RX_MODE();
    delay_1ms(10);
}
```

同时检查 `RS485_SendData()` 实现：

```c
void RS485_SendData(const uint8_t* data, uint16_t len)
{
    uint16_t i;
    if ((data == NULL) || (len == 0U)) return;

    RS485_TX_MODE();  /* 切换到发送模式 */
    
    for (i = 0U; i < len; i++) {
        usart_data_transmit(RS485_USART, data[i]);
        while (RESET == usart_flag_get(RS485_USART, USART_FLAG_TBE));
    }
    
    /* ✅ 这一步很关键：等待最后一个字节真的发完 */
    while (RESET == usart_flag_get(RS485_USART, USART_FLAG_TC));
    
    /* ✅ 这一步也很关键：立即切回接收模式 */
    RS485_RX_MODE();
}
```
